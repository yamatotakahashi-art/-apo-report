# 営業報告ツール 要件定義（AlphaDrive 案ベース）

> 参考デザイン: [`docs/reference/AlphaDrive_目標デザイン.dc.html`](./reference/AlphaDrive_目標デザイン.dc.html)
> （Claude のデザインプロトタイプ。これを **目標仕様** として、既存のログイン版 webapp の土台の上に作り直す）

## 0. この文書の位置づけ

- 今の単一HTML版（`index.html` / `demo/` / `webapp/public/legacy/index.html`）は **動くまま残す**。
- 新版は `webapp/`（Next.js + Auth.js + Neon）の土台を **活かして** 作り直す。土台は捨てない。
- 確定済みの2大方針：
  1. **AlphaDrive デザインを目標仕様として作り直す**（iframe の中の1ファイルに継ぎ足すのではなく、ちゃんとした Next.js のページとして組む）。
  2. **Gmail/Slack へ API で自動投稿はしない**。読み取った情報を「型（テンプレート）」に流し込んで整形テキストを作り、**本人がコピーして送る**。＝今の機密安全モデルを維持する。

## 0.1 業務の前提（用語）

- 自社は **営業代行**。**依頼主（発注元）** から委託を受け、**エンド顧客**へ架電・メールする。
- **エンド顧客** … メールの送付先。氏名・会社・電話・メール等は最も保護すべき顧客データ。
- **依頼主（発注元）** … 営業を委託してきた側のお客様。案件ごとに固定の取引先で、CC に入ることがある（エンド顧客ではない）。

## 1. 大方針・セキュリティ原則（最優先）

| 原則 | 中身 |
|------|------|
| エンド顧客データはブラウザ内のみ | 会社名・電話・住所・顧客メール・担当者名・架電メモは **localStorage と画面の中だけ**。Neon にも外部サーバにも送らない。 |
| サーバに置くのは「設定」だけ | ユーザー/役割、案件設定（チャンネル名・メンション・CC・テンプレ文面）、メンバー。これらはエンド顧客データではない。 |
| 自動投稿しない | パース結果を型に流し込み、整形テキストを生成 → **本人がコピーして送る**。サーバに認証情報を預けて自動送信、はやらない。 |
| 送信前の伏せ字は維持 | フィードバック（バグ/要望）を Slack に送る機能は従来どおり送信前に `redactCustomerData()` で伏せ字化。 |
| アクセス制限は既存のまま | 社内ドメイン限定の Google ログイン、テンプレ編集は staff のみ（サーバ側で強制）。 |
| 共有が必要な顧客データは IT 部門の承認形式に従う | 履歴共有・上長確認は顧客データの共有を伴う。**最も安全な方式をシステム部門に打診**し、その承認形式に従って実装する（§8）。 |

## 2. 画面構成（サイドバー＋5画面）

### 2.1 📮 報告（メイン）
- **目的**：1件の架電結果を、Slack 報告文＋メール下書きの「型」に流し込む。
- **入力**：案件セレクタ／Sheet 行ペースト → 自動検出（会社名・電話・住所・メール）。
- **outcome**：`アポ獲得` / `資料請求` の2種のみ（不在・受付NG等は対象外＝teleapo 側）。
- **追加項目**：
  - 資料請求 → 宛先メール・宛名/事業部・送付資料・次回連絡打診
  - アポ獲得 → 宛先メール・面談日時・面談リンク・次回連絡打診
- **右ペイン**：Slack 投稿プレビュー＋メール下書きプレビュー（リアルタイム）。
- **出力＝コピーのみ**：型に流し込んだ整形テキストを **コピー** するだけ（Gmail/Slack の自動オープン・自動投稿はしない。セキュリティ面の不安を避ける）。
- **任意：上長へ確認依頼**（§6 参照／ソフト承認・必須ではない）。
- **データの出所**：エンド顧客データ＝ブラウザ。テンプレ・案件設定＝サーバ。

### 2.2 📁 案件一覧
- 案件カード：名前・説明・Slack チャンネル名・メンバー・今月件数（アポ/資料の内訳）。
- アクティブ / アーカイブ切替、`+ 新規案件`。
- **件数の集計は非PII**（日付×案件×outcome の件数のみ）→ サーバ集計でも安全。会社名など明細はここに出さない。

### 2.3 案件編集（タブ：基本 / Slack / メールテンプレ / 資料 / メンバー）
- **Slack タブ**：投稿先チャンネル・メンション対象・投稿テンプレ（差し込みタグ対応）・利用可能タグ一覧・変更履歴。
- **メールテンプレ タブ**：`資料送付用` / `アポお礼用` 切替、`会社標準` / `個人カスタム` 切替、件名、**案件単位で固定の CC**、本文（差し込みタグ）。
  - CC は **社内アドレス＋依頼主（発注元）** を想定。エンド顧客は CC に入れない。
- **編集権限**：会社標準テンプレ・案件設定は **staff のみ**（intern は閲覧）。個人カスタムは各自のブラウザ。

### 2.4 📋 履歴
- 過去報告の一覧（日時・案件・会社名・outcome・連携状況）。
- **会社名（エンド顧客）を含む＝共有には保護が要る**。
- **共有は必須**（チームで見たい）。→ **保存/共有方式はシステム部門の承認形式に従う**（§8の打診結果で確定）。それまでは暫定でブラウザ内保存。

### 2.5 ⚙️ 設定
- 表示名・名乗り（メール冒頭）・署名。→ 各自のブラウザに保存（テンプレはツール内に保存）。
- Gmail 自動連携はしない方針。署名等のテキスト管理のみ。

## 3. データモデル（ドラフト）

### 3.1 サーバ（Neon）＝設定のみ・エンド顧客データなし
```
users(email PK, name, role['staff'|'intern'], created_at)            -- 既存
projects(id PK, name, description, slack_channel, archived, created_at)
project_members(project_id, user_email)                              -- 表示用メンバー
project_cc(project_id, email)                                        -- 社内CC＋依頼主（発注元）
project_mentions(project_id, mention)                                -- Slackメンション対象
templates(project_id, kind['slack'|'email_material'|'email_appo'],
          subject, body, updated_by, updated_at, PK(project_id,kind))
report_stats(project_id, day, outcome, count)                        -- 非PIIの件数集計のみ
```
- 既存 `templates(mail_type,status,…)` は project 単位へ拡張する（移行が要る）。
- `project_cc` の依頼主アドレスは外部の取引先メール。エンド顧客ではないが外部連絡先のため、§8 の打診対象に含める。

### 3.2 ブラウザ（localStorage）＝エンド顧客データ・個人設定
- 個人設定（表示名・名乗り・署名）
- 個人カスタムテンプレ
- 報告履歴の明細（会社名等を含む）※共有方式が確定するまでの暫定
- 入力途中のドラフト

## 4. 「型に流し込む」出力の仕組み

- **差し込みタグ**：`{company_name}`『{reporter_name}』`{mentions}` `{outcome_label}` `{outcome_icon}` `{address}` `{recipient_name}` `{attachments}` `{meeting_at}` `{meeting_url}` `{next_contact}` `{service_name}` `{service_url}` `{signature}` など。
- **Slack 出力**：案件の Slack テンプレにタグを流し込み → 整形テキスト → **コピー**（自動投稿しない）。
- **メール出力**：案件のメールテンプレに流し込み → 整形テキスト → **コピー**（Gmail 自動オープンもしない）。
- 既存の堅い実装（`redactCustomerData()`・`zen2han()`・行パース・自動検出）は **作り直しでも移植して再利用**する（ゼロから書き直さない）。

## 5. 役割（staff / intern）

| | staff（正社員） | intern |
|---|---|---|
| 報告作成・コピー | ✅ | ✅ |
| 会社標準テンプレ／案件設定の編集 | ✅ | ❌（閲覧・利用のみ） |
| 個人カスタムテンプレ | ✅ | ✅ |
| 上長へ確認依頼を出す | ✅ | ✅ |

- 役割は初回ログイン時に `ADMIN_EMAILS` を基に付与、以後は `users.role` を直接更新（既存仕様）。
- サーバ側 `PUT /api/...` で `role==='staff'` を強制（UI で隠すだけにしない）。

---

## 6. 追加機能：上長への確認依頼（任意・ソフト承認）

- **狙い**：「必ず承認」ではなく **不安解消用**。上長の工数を増やさない軽い仕組み。
- **動き（案）**：報告画面に「上長に確認を依頼」ボタン。押すと上長へ軽い通知が飛び、上長は任意で内容を見て 👍 できる。**送信のゲートにはしない**（確認待ちでも本人はコピーして送れる）。
- **データ上の注意**：上長が内容を見る＝エンド顧客データを上長と共有することになる。これは §2.4 の履歴共有と同じく、**システム部門の承認形式に従う**（§8）。承認形式が固まるまでは、共有内容を非PIIメタデータ（案件・outcome・日時）に絞る暫定運用とする。
- **役割**：誰が誰の上長か（`reports_to` 等）は後で設計。当面は「案件のメンバー/管理者」を上長候補とする。

## 7. 段階リリース計画

- **Phase 1（土台＋メイン画面）**：サイドバー＋報告画面。案件は固定1件でOK。パース→型→プレビュー→コピー。エンド顧客データはブラウザのみ。**※IT 部門の回答を待たずに着手可能**（共有を伴わないため）。
- **Phase 2（案件管理）**：projects/メンバー/CC（社内＋依頼主）/メンション/per-project テンプレ（staff 編集）。案件一覧・案件編集タブ。
- **Phase 3（履歴・確認依頼・設定・仕上げ）**：履歴・上長確認依頼（**IT 部門の承認形式に従う**）、設定（署名等）、UX 仕上げ（60秒取消トースト等）。
- 各 Phase は `demo/` 相当で固めてから本番反映、の運用は継続。今の動くアプリは新版完成まで残す。

---

## 8. システム部門への打診（顧客データ共有の方式）

履歴共有・上長確認は **エンド顧客データの共有** を伴うため、最も安全な方式をシステム部門に打診する。
打診文書：[`docs/システム部門への打診.md`](./システム部門への打診.md)

打診で確定させたいこと：
1. 報告履歴（会社名等を含む）を **チームで共有** する際の承認された保存先・暗号化・アクセス制御・保持期間。
2. 上長確認依頼で **報告内容を上長に見せる** 際の許容範囲・方式。
3. 依頼主（発注元）のCCアドレスを **設定としてサーバ保存** することの可否。

確定するまでの暫定：履歴明細はブラウザ内、確認依頼は非PIIメタデータのみ。

## 9. Googleカレンダー連携（リマインド）※追加要望

アポ／フォローの取りこぼし防止に、**リマインドをGoogleカレンダーへ登録**できるようにする。

- **前日リマインド**：面談（アポ）の前日に通知。
- **フォローコール等のリマインド**：後追い架電などの予定を登録。
- **方式（機密方針に整合）**：API自動連携はしない。**`.ics` ファイルのダウンロード**（前日アラーム `VALARM` 入り）か、**Googleカレンダーの登録リンク（プリフィル）**を生成し、**本人が自分のカレンダーへ登録**する。サーバに認証情報も顧客データも預けない。
- **データ**：日時・件名（会社名を含めるかは本人の操作。顧客データはブラウザ内で生成し、本人のカレンダーへ）。
- **置き場所**：報告画面の面談日時／次回連絡の近くに「📅 リマインドを登録」。実装は Phase 2 と並行 or Phase 3。

## 10. Phase 1 実装メモ（反映済み）

- 報告画面・サイドバー・解析→型→プレビュー→コピーを `/v2` に実装（staff限定・既存/と配布版は不変）。
- **Slack報告文から「担当（報告者名）」行は削除**（チャンネルでは投稿者が自明なため）。`ご担当`（顧客側担当）は残す。
- 状況区分は架電メモから自動判定し、メール文面テンプレを自動選択（手動変更可）。
