# 営業報告ツール 要件定義（AlphaDrive 案ベース）

> 参考デザイン: [`docs/reference/AlphaDrive_目標デザイン.dc.html`](./reference/AlphaDrive_目標デザイン.dc.html)
> （Claude のデザインプロトタイプ。これを **目標仕様** として、既存のログイン版 webapp の土台の上に作り直す）

## 0. この文書の位置づけ

- 今の単一HTML版（`index.html` / `demo/` / `webapp/public/legacy/index.html`）は **動くまま残す**。
- 新版は `webapp/`（Next.js + Auth.js + Neon）の土台を **活かして** 作り直す。土台は捨てない。
- 確定済みの2大方針：
  1. **AlphaDrive デザインを目標仕様として作り直す**（iframe の中の1ファイルに継ぎ足すのではなく、ちゃんとした Next.js のページとして組む）。
  2. **Gmail/Slack へ API で自動投稿はしない**。読み取った情報を「型（テンプレート）」に流し込んで整形テキストを作り、**本人がコピー／プリフィル画面を開いて送る**。＝今の機密安全モデルをそのまま維持する。

---

## 1. 大方針・セキュリティ原則（最優先）

| 原則 | 中身 |
|------|------|
| 顧客データはブラウザ内のみ | 会社名・電話・住所・顧客メール・担当者名・架電メモは **localStorage と画面の中だけ**。Neon にも外部サーバにも送らない。 |
| サーバに置くのは「設定」だけ | ユーザー/役割、案件設定（チャンネル名・メンション・CC・テンプレ文面）、メンバー。これらは顧客データではない。 |
| 自動投稿しない | パース結果を型に流し込み、整形テキストを生成 → 本人がコピー／Gmail 作成画面を開いて送る。サーバに認証情報を預けて自動送信、はやらない。 |
| 送信前の伏せ字は維持 | フィードバック（バグ/要望）を Slack に送る機能は従来どおり送信前に `redactCustomerData()` で伏せ字化。 |
| アクセス制限は既存のまま | 社内ドメイン限定の Google ログイン、テンプレ編集は staff のみ（サーバ側で強制）。 |

---

## 2. 画面構成（サイドバー＋5画面）

### 2.1 📮 報告（メイン）
- **目的**：1件の架電結果を、Slack 報告文＋メール下書きの「型」に流し込む。
- **入力**：案件セレクタ／Sheet 行ペースト → 自動検出（会社名・電話・住所・メール）。
- **outcome**：`アポ獲得` / `資料請求` の2種のみ（不在・受付NG等は対象外＝teleapo 側）。
- **追加項目**：
  - 資料請求 → 宛先メール・宛名/事業部・送付資料・次回連絡打診
  - アポ獲得 → 宛先メール・面談日時・面談リンク・次回連絡打診
- **右ペイン**：Slack 投稿プレビュー＋Gmail 下書きプレビュー（リアルタイム）。
- **送信ボタンの実体**：「型に流し込んだ整形テキストをコピー」＋「Gmail 作成画面を開く（プリフィル／API不使用）」。**自動投稿はしない。**
- **データの出所**：顧客データ＝ブラウザ。テンプレ・案件設定＝サーバ。

### 2.2 📁 案件一覧
- 案件カード：名前・説明・Slack チャンネル名・メンバー・今月件数（アポ/資料の内訳）。
- アクティブ / アーカイブ切替、`+ 新規案件`。
- **件数の集計は非PII**（日付×案件×outcome の件数のみ）→ サーバ集計でも安全。会社名など明細はここに出さない。

### 2.3 案件編集（タブ：基本 / Slack / メールテンプレ / 資料 / メンバー）
- **Slack タブ**：投稿先チャンネル・メンション対象・投稿テンプレ（差し込みタグ対応）・利用可能タグ一覧・変更履歴。
- **メールテンプレ タブ**：`資料送付用` / `アポお礼用` 切替、`会社標準` / `個人カスタム` 切替、件名、**案件単位で固定の CC**（社内アドレス）、本文（差し込みタグ）。
- **編集権限**：会社標準テンプレ・案件設定は **staff のみ**（intern は閲覧）。個人カスタムは各自のブラウザ。

### 2.4 📋 履歴
- 過去報告の一覧（日時・案件・会社名・outcome・連携状況）。
- **会社名を含む＝顧客データ**。→ **明細はブラウザ内（localStorage）に保存**（後述 §8 で要確認）。

### 2.5 ⚙️ 設定
- 表示名・名乗り（メール冒頭）・署名。→ 各自のブラウザに保存。
- Gmail 連携表示（自動投稿しない方針なので「連携」は将来オプション。当面は表示名/署名のみ）。

## 3. データモデル（ドラフト）

### 3.1 サーバ（Neon）＝設定のみ・顧客データなし
```
users(email PK, name, role['staff'|'intern'], created_at)            -- 既存
projects(id PK, name, description, slack_channel, archived, created_at)
project_members(project_id, user_email)                              -- 表示用メンバー
project_cc(project_id, email)                                        -- 社内CC（顧客でない）
project_mentions(project_id, mention)                                -- Slackメンション対象
templates(project_id, kind['slack'|'email_material'|'email_appo'],
          subject, body, updated_by, updated_at, PK(project_id,kind))
report_stats(project_id, day, outcome, count)                        -- 非PIIの件数集計のみ
```
※ 既存 `templates(mail_type,status,…)` は project 単位へ拡張する（移行が要る）。

### 3.2 ブラウザ（localStorage）＝顧客データ・個人設定
- 個人設定（表示名・名乗り・署名）
- 個人カスタムテンプレ
- 報告履歴の明細（会社名等を含む）
- 入力途中のドラフト

## 4. 「型に流し込む」出力の仕組み

- **差し込みタグ**：`{company_name}`『{reporter_name}』`{mentions}` `{outcome_label}` `{outcome_icon}` `{address}` `{recipient_name}` `{attachments}` `{meeting_at}` `{meeting_url}` `{next_contact}` `{service_name}` `{service_url}` `{signature}` など。
- **Slack 出力**：案件の Slack テンプレにタグを流し込み → 整形テキスト。**コピーのみ**（自動投稿しない）。
- **メール出力**：案件のメールテンプレに流し込み → 整形テキスト。**コピー**、または **Gmail 作成画面をプリフィルで開く**（`https://mail.google.com/...&su=...&body=...`、API・保存認証情報を使わない）。
- 既存の堅い実装（`redactCustomerData()`・`zen2han()`・行パース・自動検出）は **作り直しでも移植して再利用**する（ゼロから書き直さない）。

## 5. 役割（staff / intern）

| | staff（正社員） | intern |
|---|---|---|
| 報告作成・コピー・Gmail を開く | ✅ | ✅ |
| 会社標準テンプレ／案件設定の編集 | ✅ | ❌（閲覧・利用のみ） |
| 個人カスタムテンプレ | ✅ | ✅ |

- 役割は初回ログイン時に `ADMIN_EMAILS` を基に付与、以後は `users.role` を直接更新（既存仕様）。
- サーバ側 `PUT /api/...` で `role==='staff'` を強制（UI で隠すだけにしない）。

---

## 6. 既存土台：残す / 変える / 足す

- **残す**：Google ログイン＋社内ドメイン制限、staff/intern 役割、Neon、Vercel デプロイ、`redactCustomerData`/`zen2han`/パース等のロジック。
- **変える**：`page.tsx` の「iframe 1枚」→ サイドバー＋5画面の本物の Next.js ページへ。`templates` スキーマを案件単位へ拡張。
- **足す**：案件(projects)・メンバー・CC・メンション・per-project テンプレ、履歴（ブラウザ）、件数集計（非PII）、差し込みタグ・プレビュー・コピー/プリフィル出力。

---

## 7. 段階リリース計画

- **Phase 1（土台＋メイン画面）**：サイドバー＋報告画面。案件は固定1件でOK。パース→型→プレビュー→コピー。顧客データはブラウザのみ。
- **Phase 2（案件管理）**：projects/メンバー/CC/メンション/per-project テンプレ（staff 編集）。案件一覧・案件編集タブ。
- **Phase 3（履歴・設定・仕上げ）**：履歴（ブラウザ明細＋非PII集計）、設定（署名等）、Gmail プリフィルで開く、UX 仕上げ（60秒取消トースト等）。
- 各 Phase は `demo/` 相当で固めてから本番反映、の運用は継続。今の動くアプリは新版完成まで残す。

---

## 8. 未確定の決定事項（要確認）

1. **履歴の保存場所**：会社名を含むため、明細は **ブラウザ内（localStorage）** を既定とする（推奨：機密維持・端末ごと）。チームで共有・端末跨ぎが要るなら Neon 保存（＝会社名がサーバに乗る）に切替が必要。
2. **「Gmail 作成画面を開く」**：プリフィル URL は API ではないが、顧客メール/名前が **本人のブラウザ→本人の Gmail** に渡る。許容で良いか（コピーのみに留めるか）。
3. **案件の Slack チャンネル**：当面は「チャンネル名を表示してコピー文面を出す」だけ（投稿は手動）で良いか。
4. **CC 固定アドレス**：社内アドレス想定（顧客でない）。この理解で良いか。
